# Quadrant Todo Plan

## 1. 项目定位

这是一个个人使用的四象限 Todo 工具，用来管理日常事项，并通过四象限方法区分任务优先级：

- 重要且紧急
- 重要不紧急
- 不重要但紧急
- 不重要不紧急

目标不是做一个复杂团队协作平台，而是做一个启动快、好用、后续可以分享给别人使用的轻量事项工作台。

## 2. 技术路线

项目将作为 `le-web` 下的独立子项目存在，优先复用现有项目的部署和数据库方式。

- 前端：单页 Web 应用
- 部署：Vercel
- 数据库：复用当前 Supabase 项目，新建独立表
- 认证：自定义 `username + password`
- 服务端：Vercel Functions 负责登录、登出、session 校验

## 3. 认证方案

不使用 Supabase Auth 作为第一版主登录流程，而是实现一套轻量的自定义账号密码登录。

### 方案目标

- 注册和登录尽量快，不依赖邮箱验证
- 支持多用户账号体系，而不是只服务单用户
- 为后续接入邮箱、OAuth 等认证方式预留扩展空间

### 第一版实现方式

- 用户通过 `username + password` 注册和登录
- 密码不在前端比对
- 服务端负责密码哈希校验
- 服务端签发 `httpOnly` session cookie
- 前端后续请求依赖 cookie 访问受保护接口

### 安全边界

- 密码必须使用哈希存储，建议第一版用 `bcryptjs`
- 任务数据不能前端直连 Supabase 读写
- 所有私有数据访问都通过 Vercel Functions 完成
- 会话独立存储，避免单纯依赖前端本地 token

### 可扩展性设计

虽然第一版只做 `username + password`，但账号体系按“可扩展认证”设计。

后续可平滑新增：

- 邮箱绑定
- 邮箱登录
- 找回密码
- Google / GitHub OAuth
- 手机号登录

实现前提：

- 用户使用稳定的内部 `user_id`
- 任务等业务数据统一关联 `user_id`
- 认证方式和业务数据解耦

## 4. 数据模型

建议第一版至少使用三张表。

### `todo_users`

- `id`
- `username`
- `password_hash`
- `display_name`
- `email`
- `email_verified`
- `created_at`
- `updated_at`

约束：

- `username` 唯一
- `email` 可为空，但如果存在应唯一

说明：

- 第一版不强制邮箱
- 但预留 `email` 字段，方便后续加找回密码或其他认证方式

### `todo_sessions`

- `id`
- `user_id`
- `session_token_hash`
- `expires_at`
- `created_at`
- `last_seen_at`
- `user_agent`
- `ip`

说明：

- 会话单独建表，支持长期登录
- 后续可做登出失效、多设备管理、全设备退出

### `quadrant_tasks`

- `id`
- `user_id`
- `title`
- `description`
- `quadrant`
- `status`
- `due_at`
- `sort_order`
- `created_at`
- `updated_at`
- `completed_at`

枚举建议：

- `quadrant`: `q1 | q2 | q3 | q4`
- `status`: `todo | doing | done | archived`

## 5. MVP 功能范围

第一版控制在够用但不臃肿。

### 任务管理

- 新增任务
- 编辑任务
- 删除任务
- 标记完成
- 归档任务

### 四象限视图

- 按四象限展示任务
- 支持任务在象限间移动
- 第一版优先用按钮切换，不强依赖拖拽

### 聚焦信息

- 今日聚焦任务
- 即将到期任务
- 已完成任务统计

### 认证与会话

- 用户注册
- 用户登录
- 登出
- 基于 cookie 的登录态保持

## 6. 页面结构

建议第一版使用 3 个核心界面。

### 登录页

- 用户名
- 密码
- 登录
- 注册入口

### 主工作台

- 顶部快速新增任务
- 左侧四象限主面板
- 右侧今日聚焦 / 即将到期 / 统计摘要

### 任务详情编辑层

- 点击任务后展开编辑
- 修改标题、描述、象限、状态、截止时间

## 7. API 设计

建议使用 Vercel Functions 暴露以下接口。

### 认证

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 任务

- `GET /api/tasks`
- `POST /api/tasks`
- `PATCH /api/tasks/:id`
- `DELETE /api/tasks/:id`

## 8. 安全边界

这是轻量产品，但认证部分仍需要保持基本安全性。

- 密码使用 `bcrypt` 或 `argon2` 哈希存储
- cookie 使用 `httpOnly`
- 任务接口必须校验当前 session
- 所有数据读写按 `user_id` 限制
- 登录错误信息尽量统一，不暴露“用户名是否存在”
- 第一版应至少预留基础限流位置

## 9. 开发顺序

建议分 4 步推进。

### 阶段 1：项目骨架

- 创建目录结构
- 搭好 `index.html`
- 搭好 `api/` 与 `supabase/` 目录
- 加环境变量读取接口

### 阶段 2：认证

- 建表 SQL
- 注册 / 登录 / 登出接口
- session cookie
- 当前用户接口
- 前端登录页 / 注册页

### 阶段 3：任务核心

- 四象限主界面
- 增删改查
- 任务状态流转

### 阶段 4：体验优化

- 今日聚焦
- 到期提醒视觉提示
- 更好的排序与筛选
- 可选拖拽

## 10. 认证后续扩展

如果后续决定增加邮箱或其他认证方式，建议按下列顺序扩展：

1. 支持用户在设置页绑定邮箱
2. 增加邮箱登录入口
3. 增加找回密码能力
4. 再考虑第三方 OAuth

这一扩展不会破坏现有任务数据结构，因为任务只关联 `user_id`。

## 11. 需要确认的事项

在开始开发前，还需要你确认下面几项。

1. 注册方式是公开注册，还是邀请制/白名单注册
2. 第一版是否需要拖拽，如果不需要则先做按钮切换
3. 是否需要“截止日期 + 逾期高亮”
4. 是否需要“重复任务”能力
5. 是否需要移动端优先体验
